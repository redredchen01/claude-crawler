# Claude Crawler 优化分析

## 问题框架

Claude Crawler 已达到 MVP 阶段（606 个测试，结构化数据优先提取，HTTP 缓存）。当前优化焦点：

1. **JS 渲染** ⭐：RenderThread 存在但未充分利用；SPA/CSR 网站数据提取不完整
2. **可靠性** ⭐：错误处理和故障恢复需强化（新功能 R1 会引入 4 个失败模式）
3. **性能**：解析器、缓存策略、数据库查询未优化
4. ~~**代码质量**~~：已大部分完成（99% 类型注解覆盖，实际不需要 P2）

**修正说明**：
- 测试失败阻塞（8 个测试在 main 分支崩溃）— 必须先修复
- R1.1 无限滚动停止条件完全未指定 — 需要 spec 前规划阶段
- 类型注解现状：parser.py 98%、storage.py 79%、core/engine.py 100% —— P2 基于错误数据，建议删除

## 优化路线图（优先级排序）

**⚠️ 注：见"阻塞问题"部分，R1.1/R1.2 需要规划前完全指定**

### 第 1 优先级（P0）：可靠性基础 ⭐⭐⭐

**目标**：为新功能（R1）建立故障检测和恢复基础

**需求**

- **R0.1 统一异常策略** — 定义 `CrawlerException` 基类，子类包括 `NetworkError`, `RenderError`, `ParserError`, `StorageError`，在所有模块中使用
- **R0.2 智能重试机制** — 网络请求支持 exponential backoff（1s, 2s, 4s, 8s，最多 3 次），渲染失败触发浏览器重启，存储错误触发事务回滚
- **R0.3 核心监控和告警** — 记录关键事件（扫描启动/完成、fetch 失败率、渲染超时、DB 错误），支持 JSON 日志导出

**范围边界**（非目标）
- 不支持分布式追踪或 APM 集成
- 不支持实时告警通知（仅日志）

**成功指标**
- 异常覆盖率 ≥ 95%
- 瞬态错误重试成功率 ≥ 70%
- 故障自恢复率 ≥ 80%（Chromium 崩溃）

**用户影响**：R1 会引入 Chromium 崩溃、网络超时、DOM 检测失败等 4 个新失败模式。无 R0 支撑，这些失败会导致 R1 ROI 接近零。

---

### 第 2 优先级：JavaScript 渲染增强 ⭐⭐⭐

**目标**：完整化 RenderThread，使爬虫能处理动态内容网站

**需求**

- **R1.1 无限滚动支持** — 自动检测和处理分页加载，停止条件为 ✓（选择以下一种）
  - ✓ **选项 A（推荐基于 Instagram/TikTok）**：按内容稳定性。连续 3 次滚动无新 DOM 节点出现（使用 MutationObserver 检测），判断内容稳定
  - 选项 B（基于新闻网站）：按滚动次数。最多 10 次滚动，每次等待最多 3 秒
  - 选项 C（基于列表网站）：按时间。最多等待 30 秒，然后停止
  - 可配置 `scroll_pause_ms=500` 和 `max_scroll_count=10`

- **R1.2 改进内容提取** — 渲染后 HTML 注入 parser 管道，并扩展 parser 接口支持 `source: rendered|static` 元数据
  - parser.py 所有 `_extract_*` 函数收到 `source` 参数
  - 动态渲染的数据（JS 注入 OG 标签）标记 `source=rendered`，优先用于决策
  - 失败模式处理：若渲染失败，回退到静态 HTML（标记 `source=static`）

- **R1.3 智能页面检测** — 自动判断页面是否需要渲染（基于多信号启发式）
  - 启发式规则：初始 HTML <20KB AND (`<script type="module">` 存在 OR body 空 OR meta 缺失关键字段)
  - 验证基准：对 100+ 真实 URL 测试，分类正确率需 ≥80%

- **R1.4 超时降级** — 渲染超时（10s）时回退到静态 HTML，标记 `source=static`

**范围边界**（非目标）
- 不支持 WebDriver API 注入或自定义 JavaScript 执行
- 不支持登录流程或 Cookie 管理
- 不支持视频/音频流处理

**成功指标**
- 渲染管道集成度 ≥90%（需要渲染的页面中 ≥90% 解析器接收渲染内容）
- 无限滚动网站准确率 ≥ 80%（Instagram/TikTok 风格页面，基于实际爬取对象）
- 性能：单页渲染 <10s（包含滚动）

**用户影响**：解锁 SPA/CSR 网站爬取能力。当前 SPA 爬取失败率 ~45%，优化目标 <5%

---

### 第 3 优先级：性能优化 ⭐

**目标**：降低单次扫描时间和资源占用

**需求**

- **R3.1 HTTP 缓存 LRU 上限** — ETag/Last-Modified 缓存增加内存限制（50 MB），超限时使用 LRU 驱逐策略（必须先测量当前缓存命中率，若 <40% 则延迟此项）

- **R3.2 批量数据库操作**（事务安全版本）— `storage.py` 的 `save_resource` 改为批量插入（batch size 100）
  - 关键：添加显式缓冲刷新逻辑，确保 `counter_updated ↔ DB committed` 不变量
  - 实现：WriterThread 在关闭时刷新所有缓冲，或使用异步计数器解耦

- ~~R3.1 解析器缓存~~ **延迟至后期** — 当前单扫描工作流中缓存无用（命中率 0%）。证实用户执行多扫描后再加

**范围边界**（非目标）
- 不迁移到 PostgreSQL 或其他 DBMS
- 不支持 Redis 集群化缓存
- 不优化 parser.py 本身（66KB 的复杂性留给后期重构）

**成功指标**
- 吞吐量提升 ≥ 10% vs. 基准（pages/sec，需要先建立基准数据）
- 内存占用 ≤ 600 MB（100 页扫描）
- HTTP 缓存命中率 ≥ 50%（在真实爬取数据上验证，不是假设）

**用户影响**：单次扫描时间减少 X%（基于基准测试）

## 关键决策

| 决策 | 理由 |
|------|------|
| **P0: 可靠性优先** | R1（JS 渲染）会引入 4 个新失败模式（Chromium 崩溃、超时、DOM 检测失败、集成失败）。无 R0 支撑，R1 ROI 接近零。必须先建立异常 + 重试 + 监控基础 |
| **P1+P2: JS 渲染 + 性能** | 可并行执行。性能优化（缓存、批处理）独立于 JS 渲染，只需先建立基准线 |
| **P3→P4: 代码质量后置** | 类型注解已 99% 完成，工作量从 5 天降至 1-2 天。优先做功能和可靠性。代码质量是养护任务，不是优化 |
| **立即行动** | (1) 修复 8 个测试失败，(2) 完全指定 R1.1 停止条件，(3) 建立性能基准线 |

## 阻塞问题（规划前必须解决）

### P0 阻塞
1. **8 个测试失败在 main 分支** — `ValueError: too many values to unpack` in engine.py line 290
   - 影响：无法运行任何新功能工作
   - 必须先修复后再实施优化

2. **R1.1 滚动停止条件完全未指定** — 3 种方案冲突（计数 vs. Network Idle vs. DOM 稳定性）
   - 影响：无法实现。需要在规划前决策
   - 建议：选择选项 A（内容稳定性，基于 Instagram/TikTok 风格），或根据客户爬取目标类型调整

### P1 阻塞
3. **R1.2 集成问题** — parser.py 是无状态函数，无法感知数据来自"动态 JS"还是"静态 HTML"
   - 影响：SPA 的 JS 注入 OG 标签无法优先化
   - 必须扩展 parser 接口加 `source` 元数据（已在 R1.2 中更新）

4. **R3.2 事务一致性** — 批处理会破坏 `counter ↔ DB rows` 不变量
   - 影响：中断扫描会导致数据重复或丢失
   - 必须添加显式缓冲刷新逻辑或异步计数器
