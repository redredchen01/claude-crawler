# SEO Crawler Phase 4 — 数据洞察优化与本地分析引擎

## Problem Frame

**当前状态：** SEO Crawler MVP 完成了关键词爬取、结果存储、CSV 导出的基础功能。

**用户痛点：** 个人 SEO 从业者需要的不仅是原始数据，而是**可操作的洞察**：
- 哪些关键词最容易优化？（难度评分）
- 真实竞争对手是谁？（SERP 分析）
- 应该先优化哪些词？（ROI 排序）
- 这批词的大趋势如何？（趋势识别）
- 这些词之间有什么聚类关系？（主题分组）

**制约条件：** 不使用外部付费 API（GSC、Ahrefs、Semrush），在现有的搜索结果数据上进行本地分析计算。

**目标用户：** 个人 SEO 从业者、小型博客、内容营销团队（无企业级协作需求）

---

## Requirements

### **核心分析能力**

- **R1. 关键词难度评分（0-100）** — 基于 SERP 特征本地计算：
  - 搜索结果数量（结果多 = 难度高）
  - Top 10 域名权重启发式评估（品牌域名出现次数）
  - AdWords 竞价难度指标（如果有）
  - 综合评分范围：0-20 易，21-50 中等，51-80 困难，81-100 极难

- **R2. 竞争对手识别** — SERP 真实竞争对手分析：
  - 自动识别 Top 10 排名网站（竞争对手清单）
  - 展示每个竞争对手的排名位置、域名权重、内容类型
  - 支持用户筛选（只看竞争对手、排除大品牌等）

- **R3. 关键词 ROI 推荐排序** — 智能排序帮助优先级判断：
  - 维度：难度（低优先）× 搜索量估计（高优先）
  - 计算 ROI 评分 = (搜索量估计 / 难度) × 竞争对手数
  - 推荐 Top N 最高 ROI 的关键词（批量优化建议）

- **R4. 趋势分析与机会识别** — 历史对比：
  - 同一关键词多次爬取时，对比排名、竞争对手变化
  - 识别上升趋势（新机会）、下降趋势（逐渐饱和）
  - 标签：🔥 新机会（难度↓）、📈 上升（排名竞争↑）、⬇️ 饱和（竞争对手↑）

- **R5. 关键词分群与主题聚类** — 自动整理关键词：
  - 基于语义相似度、搜索意图相同性进行分群
  - 展示每个分群的特征（平均难度、平均搜索量、共同竞争对手）
  - 支持按分群优化建议（一个集中优化策略覆盖多个相关词）

### **数据可视化与导出**

- **R6. 洞察仪表板** — 可视化总览：
  - 关键词难度分布（直方图）
  - ROI Top 10 排行表
  - 竞争对手热力图（出现频率）
  - 趋势标签分布（新机会 vs 饱和）

- **R7. 增强型 CSV 导出** — 丰富的导出数据：
  - 原有列（关键词、来源、意图、排名）+ 新增：难度、ROI 评分、推荐优先级、所属分群、趋势标签

- **R8. 批量分析报告** — 生成可分享的摘要：
  - 包含：总体难度分布、Top 10 ROI、推荐优化词库、竞争对手总结
  - 支持导出为 HTML/Markdown（便于分享或嵌入博客）

### **交互与工作流**

- **R9. 快速筛选与对比** — 任务结果页增强：
  - 按难度范围筛选（简单、中等、困难）
  - 按 ROI 范围筛选
  - 按分群筛选
  - 对比两个任务的结果（哪些词机会改善了）

- **R10. 关键词深度卡片** — 单词详情视图：
  - 展示：难度、ROI、Top 5 竞争对手、趋势、所属分群
  - 支持标记用户笔记（"已优化"、"计划中"等）

## Success Criteria

- ✅ 用户可以在任务完成后立即看到关键词难度、ROI 排序、分群
- ✅ 通过 ROI 排序和分群，用户可以快速确定优先优化的 50 个词（而不是盲目选择）
- ✅ 对比历史任务，用户可以看到关键词难度是否下降（优化是否有效）
- ✅ 洞察功能全部**无需第三方付费 API**，基于本地爬虫数据和算法计算
- ✅ 导出的报告包含足够的洞察信息，用户可直接用于内容优化决策

## Scope Boundaries

### 不包含（Phase 4 范围外）

- **外部数据源整合** — GSC、Ahrefs、Semrush 等付费工具接入 → 留到 Phase 5
- **自动化与定时任务** — 后续阶段考虑（定时爬取、邮件通知）
- **协作与多用户** — 超出当前用户（个人/小团队）范围 → 未来扩展
- **高级机器学习** — 不使用复杂 ML 模型，优先用启发式算法实现
- **实时排名监控** — 聚焦批量分析，不包括持续排名跟踪

## Key Decisions

### **1. 难度评分算法——启发式方案，无机器学习**
**决策：** 使用 SERP 特征的加权启发式公式，而非黑盒 ML 模型
```
难度评分 = (搜索结果数 normalized) × 0.4 
         + (Top10 品牌域名数) × 0.3 
         + (平均域名权重估计) × 0.3
```
**原因：** 
- 可解释性强（用户能理解为什么这个词难）
- 无需训练数据，即开即用
- 可快速迭代调整权重

### **2. 搜索量估计——本地启发式，不依赖外部 API**
**决策：** 基于搜索结果数和竞争对手数的启发式估计（关键词竞争程度越高，潜在搜索量越高）
**替代方案考虑：** 如果用户提供 Google Search Console 数据，可优化估计准确度
**原因：** 避免依赖 Keyword Planner API，让用户完全独立

### **3. 数据库扩展——加入时间序列支持**
**决策：** 现有 SQLite 添加 `task_runs` 表记录多次爬取，支持趋势计算
**原因：** 很小的数据模型改动，启用了趋势分析这个关键能力

### **4. 竞争对手定义——SERP Top 10（动态）**
**决策：** 竞争对手 = 该关键词 SERP Top 10 的网站，每次爬取重新计算
**原因：** 保持动态性；不引入"永久对手列表"的维护成本

## Dependencies / Assumptions

- **假设 1：** 当前爬虫已能稳定抓取 SERP 数据（Google/Bing），包括排名位置、域名等
- **假设 2：** 后端可以存储 SERP 快照（域名列表、搜索结果数等），支持历史查询
- **假设 3：** 用户愿意接受本地启发式评分，而不是第三方付费工具的官方评分
- **依赖：** Phase 3 前端完成后可进行（API 响应结构已定义）

## Outstanding Questions

### Resolve Before Planning

- **[R1][Product] 难度评分的权重系数（0.4/0.3/0.3）是否确定，还是需要用户反馈调整？**
  - 影响：影响难度评分准确度，可能需要在实现后 A/B 测试调整
  - 建议：先使用提议的权重上线，第一版后收集用户反馈迭代

- **[R3][Product] ROI 推荐排序中的"搜索量估计"是否足够准确？用户是否愿意自己输入真实搜索量？**
  - 影响：这是 ROI 排序的核心，准确度直接关系推荐质量
  - 建议：Phase 4 用启发式估计，如果用户提供 GSC 数据可优化；为 Phase 5 留口子

### Deferred to Planning

- **[R5][Technical]** 关键词分群的相似度算法选择（余弦相似度、编辑距离、语义embedding）
  - 建议在规划时研究，选择平衡效果与性能的方案
  
- **[R6][Technical]** 直方图、热力图的前端可视化库选择（Chart.js、Recharts、D3）
  - 依赖当前项目已使用的库；建议规划时统一

- **[R8][Needs research]** HTML 报告生成的最佳实践（是否使用 Puppeteer 服务端渲染，还是前端生成）
  - 性能和可维护性权衡，规划时研究

## Implementation Roadmap

### Phase 4.1: 基础分析引擎（难度、竞争、ROI）
- 后端：计算难度评分、竞争对手识别、ROI 排序算法
- 前端：洞察仪表板（3 个图表）、快速筛选、结果表格增强
- 预期工作量：4 个实现单元

### Phase 4.2: 趋势与分群（历史对比、语义分组）
- 后端：多次爬取历史存储、趋势识别算法、关键词分群算法
- 前端：趋势标签、分群筛选、分群详情卡片
- 预期工作量：3 个实现单元

### Phase 4.3: 高级导出与报告
- 后端：报告生成（HTML/Markdown）
- 前端：导出交互（CSV、报告、对比）
- 预期工作量：2 个实现单元

**总体规模：** ~9 个实现单元，预计 2-3 周交付（取决于平行度）
