# Claude Crawler UI 增强：实时进度统计 & 历史管理

## 问题框架

当前 Streamlit UI 的进度显示和历史扫描管理功能基础但不够充分：
- **进度显示** 仅显示页数、当前URL、经过时间，缺乏运行时性能洞察（速度、预估、失败分析）
- **历史管理** 是静态列表，无搜索、过滤或排序能力，难以快速定位特定扫描

这导致用户无法有效监控扫描质量、理解性能趋势、或管理多个历史扫描。

## 需求

### A. 实时进度统计增强

**A1. 动态性能指标**
- R1. 显示实时抓取速度（页面/秒），每 1-2 秒更新一次
- R2. 基于当前速度预估剩余完成时间，精度 ±5% 左右
- R3. 展示已失败页面数和按失败原因的分布（HTTP错误、robots.txt 阻止、渲染失败等）
- R4. 累计抓取总字节数或资源发现计数（可选，低成本增强）

**A2. 详细事件流（可选）**
- R5. 显示最近 50 条页面级事件日志（可滚动），每条包含：状态、URL、成功/失败原因、耗时
- R6. 事件日志默认隐藏（折叠），用户可点击展开查看，避免进度屏幕过载

**A3. 视觉改进**
- R7. 将指标组织为清晰的卡片或网格布局（当前是简单 3 列 metric）
- R8. 对失败进度条可视化（红色或警告样式）

### B. 历史扫描管理增强

**B1. 搜索和过滤**
- R9. 按域名或 URL 关键字搜索历史扫描（模糊匹配，实时过滤）
- R10. 按状态过滤（成功、失败、部分失败）
- R11. 按资源发现数范围过滤（例：0-10、11-50、51+）
- R12. 过滤组合使用（搜索 + 状态 + 资源范围同时有效）

**B2. 排序和视图**
- R13. 按创建时间排序（最新/最旧优先），按扫描时间排序（执行耗时）
- R14. 列表使用分页或虚拟滚动以支持大量历史扫描（100+ 条）

**B3. 数据导出**
- R15. 为每次扫描提供"下载"按钮，导出该扫描的原始扫描配置 + 元数据为 JSON
- R16. 导出内容包括：目标 URL、配置参数、扫描时间、页面列表、失败统计

### C. 架构要求

**C1. 进度流数据结构**
- R17. `ProgressCoalescer` 的事件字典扩展，包含：pages_done, pages_total, current_url, failed_count, failure_reasons (dict), fetch_speed_pps, eta_seconds, event_log (list of dicts)
- R18. 向后兼容现有事件格式（新字段可选）

**C2. 历史查询能力**
- R19. 增强存储层 (`storage.py`) 支持过滤和排序查询，例 `list_scan_jobs(db, domain_filter, status_filter, resource_range, sort_by)`
- R20. 新增导出函数 `export_scan_job_metadata(db, scan_job_id) -> dict`

## 成功标准

- 进度屏幕在扫描期间清晰显示至少 6 个动态指标（页数、速度、预估时间、失败数等），无性能滞后
- 历史列表支持组合搜索/过滤，用户可在 <100ms 内找到特定扫描（假设 100+ 扫描历史）
- UI 布局美观、文字清晰，与 Streamlit 设计语言保持一致
- 所有新功能通过单元测试覆盖（模拟存储、模拟进度事件）

## 范围边界

- **不包含** 远程日志持久化或分布式进度流（仅本地 SQLite）
- **不包含** 从历史扫描重新发起扫描快捷方式（可作为后续功能）
- **不包含** 进度通知（邮件、Slack 等）
- **不包含** 对现有爬虫引擎逻辑的改动，仅在呈现层增强

## 关键决策

**进度流数据补充方式**：通过 `engine.py` 中的进度事件回调计算（速度、预估时间）而非独立数据收集，避免重复统计。

**历史事件日志是否必需**：标记为可选（R5-R6）。如果时间允许可添加；如果时间紧张可推迟到 Phase 2。

**导出格式**：JSON 而非 CSV，便于后续处理和重新导入（如有需要）。

## 依赖 / 假设

- `storage.py` 已支持基本的 CRUD，新增过滤逻辑假设 SQLite 查询不会成为性能瓶颈（<1K 扫描历史）
- `ProgressCoalescer` 性能足以容纳新增字段（预期信息量增加 <10%）
- Streamlit 应用重新运行（rerun）频率控制在合理范围（当前 1 秒一次）
